# Refinery Procurement System

A Rust-based procurement management system for refinery operations, implementing a multi-service architecture with domain-driven design principles.

## Project Structure

This is a **Cargo workspace** project with multiple services sharing common libraries:

### Applications (`apps/`)
Four separate services that share common libraries:

```
apps/
├── admin/          # Admin service - supplier & system management
├── auth/           # Authentication service - login & user management
├── catalog/        # Catalog service - refinery equipment catalog
├── procurement/    # Procurement service - purchase order management
└── worker/         # Background worker - async tasks & message processing
```

### Libraries (`libs/`)
Shared libraries containing business logic and data access:

```
libs/
├── domain/         # Business logic, services, repositories, DTOs
└── database/       # Database models, schema, and migrations
```

## What Each Layer Does

### **Apps Layer** - The Entry Points

- **Admin Service**: Administrative operations
  - Supplier management (CRUD operations)
  - System operations (catalog seeding)
  - Audit log viewing
  - Swagger UI at `/swagger-ui`

- **Auth Service**: Authentication and authorization
  - User login/logout
  - JWT token management
  - User profile endpoints
  - Swagger UI at `/swagger-ui`

- **Catalog Service**: Equipment catalog management
  - Catalog item listing and search
  - Item compatibility checking
  - Secondary ID lookups
  - Swagger UI at `/swagger-ui`

- **Procurement Service**: Purchase order management
  - Create and manage purchase orders
  - Add items to orders
  - Submit orders for approval
  - Order status tracking
  - Swagger UI at `/swagger-ui`

- **Worker Service**: Background processing
  - RabbitMQ message consumption
  - Async task execution
  - Scheduled jobs

### **Domain Layer** - The Brain

Where all the business logic happens:

- **Services** (`libs/domain/src/services/`):
  - Implement business rules and workflows
  - Coordinate between repositories
  - Handle complex operations like order creation, authentication
  - Examples: `PurchaseOrderService`, `AuthService`, `CatalogService`, `SupplierService`

- **Repositories** (`libs/domain/src/repositories/`):
  - Abstract database operations
  - Provide CRUD operations using `impl_crud_repo!` macro
  - Custom query methods with filtering and pagination
  - One repository per entity

- **DTOs** (`libs/domain/src/dto/`):
  - Data transfer objects for API requests/responses
  - Validation with serde and validator
  - OpenAPI schemas with utoipa
  - Separate internal models from external contracts

- **HTTP Controllers** (`libs/domain/src/http/controllers/`):
  - Shared controllers (e.g., `auth_controller`)
  - Middleware implementations
  - Request/response handling

### **Database Layer** - The Foundation

Manages all data persistence:

- **Models** (`libs/database/src/models/`):
  - Diesel ORM models
  - Derive `Queryable`, `Insertable`, `Identifiable`
  - Map to database tables

- **Schema** (`libs/database/src/schema.rs`):
  - Auto-generated by Diesel
  - Defines table structures
  - Used by models for type-safe queries

- **Enums** (`libs/database/src/enums/`):
  - Database enum definitions
  - Type-safe enumerations (UserRole, UserStatus, CatalogCategory, etc.)

- **Migrations** (`resources/migrations/`):
  - Version-controlled database changes
  - Migration files for all entities
  - Run with `diesel migration run`

## Key Design Patterns

### 1. **Clean Architecture**
```
Controllers → Services → Repositories → Models
   (HTTP)    (Business)   (Data Access)  (Database)
```
Each layer depends only on the layer below it, making the codebase modular and testable.

### 2. **Supervisor Pattern**
All services use `foxtive-supervisor` for process management:
- Graceful shutdown handling
- Health monitoring
- Task lifecycle management

### 3. **Multi-Service Architecture**
- Independent services for different domains
- Shared libraries for common logic
- Service-specific Swagger documentation

## Getting Started

### Prerequisites
- Rust 1.93+
- PostgreSQL
- Redis
- RabbitMQ

### Setup

1. **Clone and navigate to the project**:
```bash
cd refinery-procurement
```

2. **Configure environment**:
```bash
# Edit .env file with your configuration
# Key variables:
#   - DATABASE_URL
#   - PROC_REDIS_DSN
#   - PROC_RMQ_DSN
#   - Server ports for each service
```

3. **Run database migrations**:
```bash
diesel migration run
```

4. **Build the project**:
```bash
cargo build
```

5. **Run individual services**:
```bash
# Admin Service
cargo run --package admin-service

# Auth Service
cargo run --package auth-service

# Catalog Service
cargo run --package catalog-service

# Procurement Service
cargo run --package procurement-service

# Worker Service
cargo run --package worker-service
```

### Adding a New Entity

1. Create migration in `resources/migrations/`
2. Add model in `libs/database/src/models/`
3. Create repository in `libs/domain/src/repositories/`
4. Create service in `libs/domain/src/services/`
5. Add controller if API endpoint needed
6. Add utoipa decorators for OpenAPI documentation

### Code Style

- Models derive: `Queryable`, `Insertable`, `Identifiable`, `Serialize`
- Repositories use `impl_crud_repo!` macro
- Services handle business logic, not controllers
- All fields match between model, schema, and migration
- DTOs derive: `Deserialize`, `Serialize`, `Validate`, `ToSchema` (for OpenAPI)
- Controllers use `#[utoipa::path]` for API documentation

## Philosophy

This project follows these principles:

1. **Separation of Concerns**: Each layer has a single responsibility
2. **Type Safety**: Leverage Rust's type system for correctness
3. **Explicit Errors**: Use `AppResult` for clear error handling
4. **Convention over Configuration**: Standard patterns reduce cognitive load
5. **Graceful Degradation**: Supervisor pattern ensures reliability
6. **API-First Design**: OpenAPI/Swagger documentation generated from code
